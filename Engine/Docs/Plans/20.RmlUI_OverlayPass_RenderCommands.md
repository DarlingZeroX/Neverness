# RmlUI Overlay Pass — RenderCommands 路径集成计划

**日期**: 2026-06-19
**前置依赖**: 19.ViewportSurface_RenderCommands.md（Phase 1-4 已完成）
**审查修订**: v2（采纳架构审查 7 项反馈）

---

## 1. 背景

### 现状
- RenderCommands 路径（C# → Flat Buffer → C++ Renderer2D）已正常工作
- RmlUI Overlay Pass 在 `ViewportSurfaceRuntimeApi.cpp:628-654` 被注释掉
- 原因：`NNRmlUISystem.Tick()` 需要 C++ `NNRuntimeScene&` 扫描 `NNRmlUIDocumentComponent`，Scene 已迁移到 C# Friflo ECS
- 旧 `RenderViewport(sceneHandle)` 路径的 RmlUI 仍工作（通过 C++ NNRuntimeScene）

### 目标
在 RenderCommands 路径中启用 RmlUI Overlay Pass：
- C# 从 Friflo ECS 收集 RmlUIDocument 数据
- 通过新 RenderCommand 类型传给 C++
- C++ RmlUIRenderer 加载文档并在 FBO 上叠加渲染（alpha 混合）

### 渲染管线（目标）

```
C# Friflo ECS
  ├─ 收集 Sprite → DrawSpriteBatch 命令
  ├─ 收集 RmlUIDocument → SetRmlDocuments 命令（纯路径，无 Guid）
  └─ 一起发给 C++

C++ 解析 Commands
  ├─ Renderer2D (World Pass) → FBO
  ├─ RmlUIRenderer.Sync(drawItems)  ← 直接传 Path，无需 Resolver
  ├─ RmlUIRenderer.Update()
  ├─ RmlUIRenderer.RenderOverlayOnScene → FBO（alpha 混合叠加）
  └─ CopyTexture (FBO → SwapChain) → Present
```

---

## 2. 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| DrawList 来源 | C# 构建，跳过 NNRmlUISystem | Scene 在 C#，没必要绕回 C++ 扫描 ECS |
| 资产路径 | C# 直接传路径字符串，**不传 Guid** | Guid 无实际意义，Path 才是加载依据 |
| AssetResolver | **删除 StringPathResolver**，Sync 直接传 Path | 避免 Guid→Path 映射的生命周期问题 |
| 增量同步 | Renderer 内部做 Entity→Runtime 三路 Diff | 文档只加载一次，不会每帧重复 LoadDocument |
| ViewTarget | 现在就支持 Scene/Game/Both | 避免 viewTarget 成为死数据 |
| 输入事件 | 保持 ViewportRender.ProcessRmlUIInput | 输入需要低延迟，不绑定渲染帧时序 |
| viewportId | 预留字段，未来多 Viewport 支持 | 输入上下文 ≠ 渲染上下文时需要区分 |
| 数据收集 | 直接 Friflo Query，不查 IInspectorService | Inspector 是编辑器，Render 是运行时，不应耦合 |

## 3. 数据结构设计

### 3.1 新增命令类型

```c
// RenderCommands.h
NN_RENDER_COMMAND_SET_RML_DOCUMENTS = 0x20
```

### 3.2 RmlDocumentEntry（276 bytes）

```c
typedef struct NNRmlDocumentEntry
{
    char     assetPath[256];    // VFS 路径（UTF-8，NUL 终结，最长 255 字符）
    int32_t  sortOrder;         // 4 bytes - 渲染排序（小的在前）
    uint32_t viewTarget;        // 4 bytes - NNRmlUIViewTarget (0=Scene, 1=Game, 2=Both)
    uint32_t entityHandle;      // 4 bytes - C# entity ID（用于 Renderer 内部 Diff）
    uint32_t viewportId;        // 4 bytes - 预留：视口 ID（0=默认）
} NNRmlDocumentEntry;           // 总计 276 bytes
```

**与 v1 对比**：
- ~~assetGuid (16B)~~ → 删除（Path 才是加载依据，Guid 无实际意义）
- ~~assetPath[480]~~ → assetPath[256]（255 字符已足够覆盖任何 VFS 路径）
- ~~reserved (4B)~~ → viewportId（预留多视口支持）
- 512 → 276 bytes（节省 46%）

### 3.3 SetRmlDocuments 命令

```c
typedef struct NNRmlDocumentsData
{
    uint32_t documentCount;     // 4 bytes
    uint32_t reserved0;         // 4 bytes
    uint32_t reserved1;         // 4 bytes
    uint32_t reserved2;         // 4 bytes
    NNRmlDocumentEntry entries[]; // 变长数组
} NNRmlDocumentsData;           // header 16 bytes + N * 276 bytes
```

总命令大小 = 8 (cmd header) + 16 (data header) + N * 276

### 3.4 C# 端对齐结构体

```csharp
[StructLayout(LayoutKind.Sequential, Pack = 4)]
public unsafe struct RmlDocumentEntry
{
    public fixed byte AssetPath[256];   // UTF-8 NUL-terminated
    public int SortOrder;
    public uint ViewTarget;
    public uint EntityHandle;
    public uint ViewportId;
}
```

## 4. C++ 端实现

### 4.1 ~~StringPathResolver~~ → 删除

v1 设计的 `StringPathResolver` 存在生命周期问题：
- `Sync()` 内部缓存 `DocumentRuntime`，可能引用 Resolver 路径
- 每帧 `Clear()` 后如果触发 `Reload/HotReload`，路径失效

**改为**：`RmlUIRenderer::Sync()` 直接接收带路径的 DrawItem，不经过 IAssetResolver。

### 4.2 RmlDrawItem 扩展

```cpp
// NNRmlUISystem.h 或新头文件
struct RmlDrawItem
{
    NNEntity              entity = 0;
    std::string           assetPath;     // VFS 路径（直接从 C# 传入）
    std::int32_t          sortOrder = 0;
    NNRmlUIViewTarget     viewTarget = NNRmlUIViewTarget::Both;
    std::uint32_t         viewportId = 0;
};
```

RmlUIRenderer::Sync / LoadDocument 改为直接使用 `item.assetPath`，不经过 IAssetResolver。

### 4.3 增量同步（三路 Diff）

RmlUIRenderer 内部维护 `unordered_map<NNEntity, RmlDocumentRuntime>`。

`Sync()` 执行三路 Diff：
```
1. 遍历新 drawList，收集 active entity 集合
2. 删除：不在 drawList 中的旧 Document → Close()
3. 新增：不在 m_Documents 中的新 entity → LoadDocument(path)
4. 修改：已存在的 entity 检查 sortOrder/viewTarget 变化 → 更新属性
```

文档只在新增时 `LoadDocument()`，不会每帧重复加载。

### 4.4 命令解析

```cpp
case NN_RENDER_COMMAND_SET_RML_DOCUMENTS:
{
    auto* data = reinterpret_cast<const NNRmlDocumentsData*>(
        cmdPtr + sizeof(NNRenderCommandHeader));

    for (uint32_t d = 0; d < data->documentCount; ++d)
    {
        const auto& entry = data->entries[d];
        NN::Runtime::RmlUI::RmlDrawItem item;
        item.entity = static_cast<NNEntity>(entry.entityHandle);
        item.assetPath = entry.assetPath;  // 直接用路径
        item.sortOrder = entry.sortOrder;
        item.viewTarget = static_cast<NNRmlUIViewTarget>(entry.viewTarget);
        item.viewportId = entry.viewportId;
        rmlDrawItems.push_back(std::move(item));
    }
    break;
}
```

### 4.5 RmlUI Overlay Pass

```cpp
auto* rmlRenderer = GetRmlUIRenderer();
if (rmlRenderer && !rmlDrawItems.empty())
{
    rmlRenderer->SetViewport(width, height);
    rmlRenderer->Sync(rmlDrawItems);    // 三路 Diff，只加载新文档
    rmlRenderer->Update();               // Context::Update

    rmlRenderer->RenderOverlayOnScene(
        rmlDrawItems,
        NN::Runtime::Scene::NNRmlUIViewTarget::Scene,
        g_CommandsFBO->GetColorRTV(),
        g_CommandsFBO->GetDepthDSV(),
        width, height);
}
```

### 4.6 ViewTarget 支持

现在就按 viewTarget 过滤渲染：
- `Scene` → 仅 Scene View
- `Game` → 仅 Game View
- `Both` → 两者都渲染

`RenderOverlayOnScene` 已有 viewTarget 参数，无需改动。

## 5. C# 端实现

### 5.1 RenderCommandBuffer 扩展

```csharp
public void AddSetRmlDocuments(ReadOnlySpan<RmlDocumentEntry> entries)
{
    int totalSize = 16 + entries.Length * 276;
    WriteCommandHeader((uint)RenderCommandType.SetRmlDocuments, totalSize);
    WriteUInt((uint)entries.Length);
    WriteUInt(0); WriteUInt(0); WriteUInt(0);
    for (int i = 0; i < entries.Length; i++)
        WriteRmlDocumentEntry(entries[i]);
}
```

## 10. 已知限制与未来扩展

1. **路径长度限制 255 字符**：足够覆盖 VFS 路径（Unity 项目最长路径约 50 字符）
2. **RmlUI 输入**：保持现有 `ProcessRmlUIInput` 路径，不通过 RenderCommand 传入

6. **PlayMode**：当前只支持 `NNRmlUIViewTarget::Scene`，PlayMode 的 Game View 渲染需要后续扩展

## 11. 关键设计决策记录

### 为什么删除 assetGuid + StringPathResolver？

v1 设计中 `assetGuid` 只是 Map Key，真正加载用的是 `assetPath`。
`StringPathResolver` 每帧 `Clear()` 存在生命周期风险——`Sync()` 内部缓存的 `DocumentRuntime` 可能引用 Resolver 路径。
改为 `Sync()` 直接接收带路径的 DrawItem，彻底消除 Guid→Path 映射。

### 为什么路径从 480 缩到 256？

Unity 项目中最长路径约 50 字符。255 字符已足够覆盖任何 VFS 路径。
节省 224 字节/条目，100 个 UI 文档节省 22KB。

### 为什么现在就支持 ViewTarget？

结构体已有 viewTarget 字段，如果渲染时硬编码 `Scene`，字段就是死数据。
现在就按 viewTarget 过滤，避免后续返工。

### 为什么加 viewportId？

未来多 Viewport（SceneViewA/B/GameView）时，输入上下文 ≠ 渲染上下文。
viewportId 让输入和渲染都能对应到同一个 Rml Context。

### 为什么直接 Friflo Query 而不查 IInspectorService？

Inspector 是编辑器概念，Render 是运行时概念。
Headless / Player / Dedicated Server 不应该依赖 Inspector 服务。
直接 Friflo Query 更干净。

### 为什么输入不走 RenderCommand？

输入事件需要低延迟（影响 hover、focus、Context::Update）。
渲染数据可以批量提交，有延迟可接受。
把输入塞进 RenderCommand 会绑定处理时机到渲染帧，引入不必要的延迟。

## 12. 实施后修复记录

### 12.1 RmlDocumentEntry 大小计算错误

**错误**: `命令 3 数据越界 (type=0x20 size=300 offset=360 remain=296 commandsSize=656)`

**根因**: `NN_RML_DOCUMENT_ENTRY_SIZE` 定义为 276，但实际结构体是 272 字节（256 path + 4×4 fields）。差 4 字节导致 C++ 解析 SetRmlDocuments 时 header 里写的 size=300（24+276）比实际 buffer 数据（24+272=296）大 4 字节，触发越界。

**修复**: C++ 宏和 C# 常量同步改为 272。

**教训**：ABI 结构体大小必须手动计算验证，不能凭印象写。

---

### 12.2 RmlUIRenderer 未初始化

**错误**: `GetRmlUIRenderer()` 返回 null，RmlUI Overlay Pass 被跳过。

**根因**: `EnsureCommandsRenderer()` 只初始化 Renderer2D + FBO，没有初始化 RmlUIRenderer。RmlUIRenderer 在 `EnsureSceneRenderer()`（旧 `RenderViewport` 路径）中创建，RenderCommands 路径从未调用过。

**修复**:
- `RmlUISingletonAccess.h` 新增 `EnsureViewportRenderInitialized()` 声明
- `ViewportRenderRuntimeApi.cpp` 实现该函数（委托给 `EnsureSceneRenderer()`）
- `ViewportSurfaceRuntimeApi.cpp` 在 `rt_viewportSurface_renderViewportCommands` 开头调用 `EnsureViewportRenderInitialized()`

**教训**：跨翻译单元的单例初始化需要显式的 ensure 函数，不能假设调用方会先触发旧路径。

---

### 12.3 拖拽 HTML 资产后 Inspector 不更新

**错误**: 拖拽 HTML 资产到 RmlUIDocumentInspector 后，GUID 标签不刷新。

**根因**: `ApplyDocumentAsset` 只更新了 ECS 数据，没有更新 UI 上的 GUID TextBlock。

**修复**: 将 `guidLabel` 作为参数传入回调，拖拽成功后同步更新 `guidLabel.Text`。
