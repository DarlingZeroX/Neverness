# 资产注册表合并计划（NNAssetRegistry + AssetRegistrySubsystem → NNRuntimeAsset）

> 日期：2026-05-31
> 模块：NNRuntimeAsset / NNRuntimeEngine / NNRuntimeEngineServices
> 状态：已实施

---

## 背景

项目中存在两套资产注册表：

| 注册表 | 模块 | API 数量 | GUID 前缀 | 状态 |
|--------|------|----------|-----------|------|
| `NNAssetRegistry` | NNAssetRegistry（独立模块） | 16 | `'NNAS'` (0x4E4E4153) | 新，活跃 |
| `AssetRegistrySubsystem` | NNRuntimeEngine（嵌在 NNEngineRuntime 中） | 8 | `'VGAS'` (0x56474153) | 旧，已弃用 |

两者功能重复，且存在以下问题：
1. **跨模块单例复制**：`NNAssetRegistry` 是 STATIC 库，每个 DLL 各有一份实例，数据不共享
2. **C ABI 转发分裂**：原始 8 个函数转发到旧注册表，新增 8 个函数转发到新注册表
3. **AssetSubsystem 空壳**：`LoadTexture`/`LoadAudio` 返回 0，完全未使用
4. **Editor 资产同步问题**：C# `EditorAssetDatabase` 同步到新注册表，但原始 C ABI 函数读的是旧注册表

---

## 合并目标

1. 将 `NNAssetRegistry` 模块的代码迁移到 `NNRuntimeAsset` 模块
2. `NNRuntimeAsset` 从 `STATIC` 改为 `SHARED`（解决跨 DLL 单例问题）
3. 删除 `NNAssetRegistry` 模块目录
4. 删除 `NNRuntimeEngine` 中的 `AssetRegistrySubsystem` 和 `AssetSubsystem`
5. 统一所有 16 个 C ABI 转发到单一注册表

## 关键设计决策

### 1. 为什么用 SHARED 而不是 STATIC？

`NNAssetRegistry::Instance()` 使用静态局部变量实现单例。在 STATIC 库中，每个链接该库的 DLL 都会有一份独立的实例，导致数据不共享。改为 SHARED 库后，单例实例在 DLL 内全局唯一，所有调用者共享同一份数据。

### 2. 为什么合并到 NNRuntimeAsset 而不是新建模块？

`NNRuntimeAsset` 已经是资产生命周期管理模块（AssetManager、Cache、Streaming），注册表（GUID ↔ Path 映射 + 依赖图）是资产系统的自然组成部分。合并后减少一个独立模块，降低构建复杂度和跨模块依赖。

### 3. DLL export 宏设计

```
NN_ASSET_DYNAMIC     — CMakeLists.txt 中 PUBLIC 定义，标记为 SHARED 库
NN_ASSET_DLL_EXPORT  — CMakeLists.txt 中 PRIVATE 定义，编译 DLL 时生效
NN_ASSET_API         — 头文件中使用，解析为 __declspec(dllexport/dllimport)
```

所有被其他模块使用的类（`NNAssetRegistry`、`NNGuidTable`、`NNDependencyTable`）都需要 `NN_ASSET_API` 宏。
